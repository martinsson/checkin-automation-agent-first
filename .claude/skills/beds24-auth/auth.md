# Beds24 v2 Authentication

## Token model

| Token | Lifetime | Where used | How obtained |
|---|---|---|---|
| **Invite code** | Single-use, expires in days | Header on `/authentication/setup` only | Generated in UI: *Settings → Account Access → API → Generate invite code* |
| **Refresh token** | Sliding 30 days (renewed on each use) | Header `refreshToken:` on `GET /authentication/token` only | Returned by `/authentication/setup` |
| **Access token** | 24h fixed | Header `token:` on **every other** API call | Returned by `GET /authentication/token` |

## Bootstrap (one-time)

1. UI → generate invite code with the scopes you need (see Scopes below).
2. Exchange immediately — invite codes are single-use:
   ```bash
   curl -H "code: $INVITE_CODE" -H "deviceName: my-tool" \
        https://api.beds24.com/v2/authentication/setup
   ```
   Response:
   ```json
   {"token": "<access>", "expiresIn": 86400, "refreshToken": "<refresh>"}
   ```
3. **Save the refresh token to your secret store immediately.** Beds24 hides it after creation — the UI list only shows a truncated prefix (`a54...`) and there's no way to retrieve the full value later. If you lose it you have to mint a new one.

## Refresh access tokens (every 24h, or per-call)

```bash
curl -H "refreshToken: $BEDS24_REFRESH_TOKEN" \
     https://api.beds24.com/v2/authentication/token
```
Response: `{token, expiresIn: 86400}`. Each call also resets the refresh token's 30-day sliding-window expiry — so a refresh token used at least monthly never dies.

## Use access tokens on every other endpoint

```bash
curl -H "token: $BEDS24_ACCESS_TOKEN" https://api.beds24.com/v2/bookings?id=...
```

## The "Token not valid" trap (401)

`{"success":false,"type":"error","code":401,"error":"Token not valid"}`

Almost always means: a **refresh token was sent as `token:`** on a non-auth endpoint. Beds24 only accepts refresh tokens on `/authentication/token` via the `refreshToken:` header. Anywhere else, only access tokens work.

This is the default failure mode when you paste the long-life token into:
- Make's HTTP module with "API Key Auth" keychain pointing at the refresh token — Make injects it as `token:` header → 401.
- n8n's "Header Auth" with `token: <refresh>` → 401.
- Anything generated by chat assistants that conflated the two tokens.

### Fix in Make / n8n / generic HTTP

Add a small step **before** any data call that fetches a fresh access token, then reference its output in the `token` header of the real call:

```
[Trigger]
   │
   ▼
[HTTP: GET /v2/authentication/token]      ← header: refreshToken: <refresh>
   │                                         outputs: { token: "...", expiresIn: 86400 }
   ▼
[HTTP: POST /v2/bookings]                 ← header: token: {{previous.token}}
```

Better long-term: cache the access token in a data store with a TTL just under 24h so you only refresh once a day instead of per execution.

Best long-term: use the platform's **official Beds24 app/connector** (Make, n8n, Zapier) — it handles the refresh dance internally. The generic HTTP route is only needed when the connector lacks the action you want.

## Scopes

Set scopes when generating the invite code. Common patterns:

| Use case | Scopes |
|---|---|
| Read bookings only | `read:bookings` |
| Read + write bookings (typical automation) | `read:bookings + write:bookings` |
| Per-guest fields (passport scans, custom Q&A) | add `read:bookings-personal + write:bookings-personal` |
| Property/template management | add `read:properties + write:properties` |
| Channel credentials / iCal | add `read:channels + write:channels` |
| Account-level reads | add `read:accounts` |

Scopes are baked into the refresh token at creation — you can't widen them later without a new invite code → new refresh token → reconnecting the integration.

Common gotcha: write-only tokens 401 on any GET. If the integration reads a booking before writing (Make's "Get a record" / typed modules often do this transparently), grant the matching read scope or you'll see `Token not valid` from the implicit read even though your write would succeed.

## Rotation / cleanup

- *Settings → Account Access → API* (also accessible at `/control3.php?pagetype=apidetail`) lists all refresh tokens with Name, Created, Expires, Scope, Property access. Each row has a **delete** link.
- Generating a new invite code does **not** automatically invalidate prior refresh tokens — they coexist until you delete them or they expire.
- Rotate by: generate new invite → bootstrap → swap secret in integration → delete the old row.
- Each integration should have its own named refresh token (deviceName at bootstrap, visible in the UI list) so you can revoke one without affecting the others.

## Account-level fields, not per-property

Refresh tokens are account-scoped. The "Property access" column controls which properties the token can touch (defaults: "All owned" or "All owned plus linked"). For multi-account setups, mint a separate refresh token per account.
